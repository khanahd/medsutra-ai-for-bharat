# Design Document: MedSutra AI Clinical Assistant

## Overview

MedSutra AI is a clinical AI assistant system designed for Indian hospitals to enhance clinical workflows through intelligent automation while maintaining clinician control over all clinical decisions. The system operates as an assistive tool that provides suggestions, summaries, and risk assessments to improve efficiency and reduce documentation burden.

### System Purpose

The system addresses critical challenges in Indian healthcare:
- Reducing clinician documentation burden (target: 5-8 minutes to 2 minutes per patient)
- Improving early cancer detection through automated radiology analysis
- Supporting multilingual patient communication across diverse populations
- Ensuring responsible AI deployment with explainability and safety guardrails
- Enabling flexible deployment for Tier-1 to Tier-3 hospitals

### Core Capabilities

1. **Clinical Document Summarization**: Generate one-page patient snapshots from multiple clinical documents using fine-tuned LLMs
2. **Radiology Cancer Detection**: Text-based analysis of radiology reports with optional multimodal image analysis
3. **Automated Documentation**: Draft clinical documents with clinician review and approval workflow
4. **Workflow Intelligence**: Suggest next clinical steps based on patient data and clinical guidelines
5. **Multilingual Support**: Generate patient-friendly summaries in English, Hindi, and regional languages
6. **Responsible AI**: Hallucination prevention, explainability, and assistive-only operation

### Design Principles

- **Clinician-in-Control**: All AI outputs require explicit clinician approval; no autonomous clinical decisions
- **Explainability-First**: Every AI suggestion includes reasoning and evidence sources
- **Privacy-by-Design**: HIPAA and DPDP Act compliance with encryption and access controls
- **Deployment Flexibility**: Support for On-Prem, Hybrid Cloud, and Offline modes
- **Indian Population Focus**: Models trained on Indian datasets to reduce bias

### Deployment Architecture

**On-Prem Deployment:**
- All components run within hospital infrastructure
- No external network dependencies
- Full data sovereignty
- Requires hospital server hardware (GPU recommended for Vision Analyzer)

**Hybrid Cloud Deployment:**
- Edge inference for Radiology Analyzer (< 5 seconds latency)
- Cloud processing for Clinical Summarizer and Documentation Assistant
- Encrypted data transmission via TLS 1.3
- Fallback to edge-only mode if cloud unavailable

**Offline Deployment:**
- Core functionality without internet connectivity
- Lightweight models optimized for CPU inference
- Periodic synchronization for model updates when connectivity restored
- Suitable for Tier-3 hospitals with limited connectivity

### Module Responsibilities

**Clinical Summarizer:**
- Process EMR notes, lab reports, discharge summaries, referral notes, radiology text
- Generate one-page Patient_Snapshot within 10 seconds
- Perform NER and entity normalization to SNOMED_CT/ICD_10/LOINC
- Merge information from multiple documents without duplication

**Radiology Analyzer:**
- Scan radiology text reports for cancer-related terms
- Generate Cancer_Risk_Flag (Low/Medium/High) within 5 seconds
- Apply organ-specific rules (lung nodules, BI_RADS for breast, LI_RADS for liver)
- Extract and highlight suspicious terms

**Vision Analyzer (Optional):**
- Analyze CT/MRI images for suspicious regions
- Use Vision Transformer or CNN models
- Generate explainable heatmaps via Grad-CAM
- Fuse text and image embeddings for multimodal assessment
- Complete analysis within 30 seconds per scan

**Documentation Assistant:**
- Auto-draft OPD notes, discharge summaries, referral letters, insurance docs
- Generate drafts within 8 seconds
- Implement Draft → Review → Edit → Approve workflow
- Use structured templates per document type

**Workflow Engine:**
- Suggest oncology referrals for high-risk findings
- Recommend biopsies for suspicious lesions
- Generate follow-up imaging reminders
- Produce patient-friendly summaries in English, Hindi, and regional languages

**Guardrail System:**
- Validate AI statements against RAG_System
- Flag unverifiable or contradictory medical information
- Prevent hallucinated facts from appearing in final documents
- Log all flagged statements for quality review

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a system—essentially, a formal statement about what the system should do. Properties serve as the bridge between human-readable specifications and machine-verifiable correctness guarantees.*

### Property Reflection

After analyzing all acceptance criteria, I identified the following redundancies and consolidations:

**Consolidated Properties:**
- Multiple timing requirements (1.1, 2.2, 3.5, 4.2, 14.3, 14.4) can be consolidated into module-specific performance properties
- Explanation requirements (2.4, 5.4, 9.1) are redundant - one property covers all AI outputs
- Logging requirements (7.6, 8.5, 15.1, 15.2) can be consolidated into comprehensive audit trail properties
- Language generation requirements (6.1, 6.2, 6.5) can be consolidated into one multilingual property
- Approval requirements (4.5, 5.6, 8.2) are redundant - one property covers all clinical actions
- Source attribution requirements (8.1, 8.6) can be consolidated
- Ontology mapping requirements (1.5, 13.3) are redundant

**Eliminated Redundancies:**
- Property about entity extraction (13.1) is subsumed by property about extracting specific entity types (13.2)
- Property about RAG validation (7.1) is subsumed by property about preventing unvalidated facts (7.5)

### Property 1: Clinical Summarization Performance

*For any* set of clinical documents (EMR notes, lab reports, discharge summaries, referral notes, or radiology text), the Clinical_Summarizer should generate a Patient_Snapshot within 10 seconds.

**Validates: Requirements 1.1**

### Property 2: Patient Snapshot Completeness

*For any* Patient_Snapshot generated from source documents containing patient information, the snapshot should include all required sections: key complaints, past medical history, current medications, abnormal findings, and pending actions.

**Validates: Requirements 1.2**

### Property 3: Patient Snapshot Length Constraint

*For any* generated Patient_Snapshot, the output length should not exceed one page (approximately 4000 characters or 60 lines).

**Validates: Requirements 1.3**

### Property 4: Medical Entity Ontology Mapping

*For any* medical entity extracted from clinical documents (diseases, medications, procedures), the entity should be mapped to at least one standard ontology code (SNOMED_CT, ICD_10, or LOINC).

**Validates: Requirements 1.5, 13.3**

### Property 5: Information Deduplication

*For any* set of clinical documents with overlapping information, the generated Patient_Snapshot should not contain duplicate entries for the same medical fact.

**Validates: Requirements 1.6**

### Property 6: Suspicious Term Detection

*For any* radiology text report containing known cancer-related suspicious terms, the Radiology_Analyzer should identify and extract those terms.

**Validates: Requirements 2.1, 2.5**

### Property 7: Radiology Analysis Performance and Validity

*For any* radiology text report, the Radiology_Analyzer should generate a Cancer_Risk_Flag that is one of {Low, Medium, High} within 5 seconds.

**Validates: Requirements 2.2**

### Property 8: Organ-Specific Risk Assessment

*For any* radiology report mentioning specific organs (lung, breast, liver), the Radiology_Analyzer should apply the appropriate organ-specific scoring system (BI_RADS for breast, LI_RADS for liver, nodule characteristics for lung).

**Validates: Requirements 2.3**

### Property 9: Vision Analysis Output Structure

*For any* medical image (CT or MRI) analyzed by the Vision_Analyzer when enabled, the output should include a heatmap and a list of suspicious regions (possibly empty).

**Validates: Requirements 3.1, 3.3**

### Property 10: Suspicious Region Coordinate Validity

*For any* suspicious region detected in a medical image, the bounding box coordinates should be within the image dimensions.

**Validates: Requirements 3.4**

### Property 11: Vision Analysis Performance

*For any* medical image scan, the Vision_Analyzer should complete analysis within 30 seconds when enabled.

**Validates: Requirements 3.5**

### Property 12: Multimodal Information Fusion

*For any* patient case where both radiology text and medical images are provided, the risk assessment should reflect findings from both modalities.

**Validates: Requirements 3.6**

### Property 13: Document Type Generation

*For any* documentation request specifying a document type (OPD note, discharge summary, referral letter, insurance doc), the Documentation_Assistant should generate a draft of the requested type.

**Validates: Requirements 4.1**

### Property 14: Documentation Performance

*For any* documentation request, the Documentation_Assistant should generate a draft within 8 seconds.

**Validates: Requirements 4.2**

### Property 15: Document Workflow State Machine

*For any* generated document draft, it should start in Draft status and support the state transitions: Draft → Review → Edit → Approve, and cannot reach Approved status without an explicit approval action.

**Validates: Requirements 4.3, 4.5**

### Property 16: Document Section Editability

*For any* document draft section, it should be marked as editable and accept modifications before approval.

**Validates: Requirements 4.4**

### Property 17: Template Structure Conformance

*For any* generated document of a specific type, the output should conform to the structured template defined for that document type.

**Validates: Requirements 4.6**

### Property 18: Draft Data Population

*For any* information present in the Patient_Snapshot or EMR, that information should appear in the generated document draft when relevant to the document type.

**Validates: Requirements 4.7**

### Property 19: High-Risk Oncology Referral

*For any* Patient_Snapshot containing high-risk findings, the Workflow_Engine should generate an oncology referral suggestion.

**Validates: Requirements 5.1**

### Property 20: Suspicious Lesion Biopsy Recommendation

*For any* radiology analysis detecting suspicious lesions, the Workflow_Engine should generate a biopsy recommendation.

## Error Handling

### Error Categories

**Input Validation Errors:**
- Invalid document format or corrupted files
- Missing required patient identifiers
- Malformed medical images (non-DICOM, corrupted headers)
- Empty or whitespace-only clinical documents
- Unsupported document types

**Processing Errors:**
- LLM inference failures or timeouts
- NER extraction failures
- Ontology mapping failures (term not found in SNOMED/ICD-10/LOINC)
- RAG system unavailable or query failures
- Vision model inference errors

**Data Access Errors:**
- Authentication failures
- Authorization failures (insufficient permissions)
- EMR integration connection failures
- Database connection failures
- Encryption/decryption failures

**Resource Constraint Errors:**
- System capacity exceeded (> 100 concurrent users)
- Request timeout (processing exceeds time limits)
- Insufficient GPU memory for vision analysis
- Storage quota exceeded

**Compliance Errors:**
- Unverifiable medical statements detected
- Contradictory information detected
- Audit log write failures
- Encryption requirement violations

### Error Handling Strategies

**Graceful Degradation:**
```typescript
// Example: Fallback when Vision Analyzer unavailable
async function analyzeRadiology(textReport: string, image?: MedicalImage): Promise<Analysis> {
  try {
    const textAnalysis = await radiologyAnalyzer.analyzeTextReport(textReport);
    
    if (image && visionAnalyzer.isEnabled()) {
      try {
        const imageAnalysis = await visionAnalyzer.analyzeImage(image);
        return fuseMultimodal(textAnalysis, imageAnalysis);
      } catch (visionError) {
        logger.warn('Vision analysis failed, falling back to text-only', visionError);
        return textAnalysis; // Graceful degradation
      }
    }
    
    return textAnalysis;
  } catch (error) {
    logger.error('Radiology analysis failed', error);
    throw new RadiologyAnalysisError('Unable to analyze radiology report', error);
  }
}
```

**Retry with Exponential Backoff:**
```typescript
// For transient failures (network, temporary service unavailability)
async function queryRAGWithRetry(query: string, maxRetries: number = 3): Promise<RAGResult> {
  let attempt = 0;
  let delay = 1000; // Start with 1 second
  
  while (attempt < maxRetries) {
    try {
      return await ragSystem.query(query);
    } catch (error) {
      attempt++;
      if (attempt >= maxRetries) {
        throw new RAGUnavailableError('RAG system unavailable after retries', error);
      }
      await sleep(delay);
      delay *= 2; // Exponential backoff
    }
  }
}
```

**User-Facing Error Messages:**
- Never expose internal system details or stack traces to users
- Provide actionable guidance for resolution
- Log detailed error information for debugging
- Maintain clinical context in error messages

**Critical Error Handling:**
- Audit log write failures: Queue logs locally and retry, escalate if persistent
- Encryption failures: Reject operation immediately, do not process unencrypted PHI
- Authentication failures: Lock account after 5 failed attempts, require admin unlock
- Guardrail system failures: Reject AI output, require manual clinician input

**Deployment-Specific Error Handling:**

**Hybrid Cloud Mode:**
- Cloud service unavailable: Fall back to edge-only processing
- Network latency exceeds threshold: Switch to edge processing
- Data transmission failures: Queue for retry, notify user of delay

**Offline Mode:**
- Model update unavailable: Continue with current model, log for later sync
- Synchronization failures: Queue updates, retry on next connectivity window

**Structured Error Logging:**

**Monitoring and Alerting:**

- High error rates (> 5% of requests): Alert to operations team
- Guardrail system flags: Daily summary to quality officers
- Performance degradation: Alert when response times exceed thresholds by 50%

**Error Recovery Procedures:**
1. Transient failures: Automatic retry with exponential backoff
2. Service unavailable: Graceful degradation to available components
3. Data corruption: Reject input, request re-submission
4. System overload: Queue requests, provide wait time estimates
5. Compliance violations: Halt operation, require manual intervention


## Testing Strategy

### Dual Testing Approach

The MedSutra system requires both unit testing and property-based testing to ensure comprehensive coverage:

**Unit Tests** focus on:
- Specific examples demonstrating correct behavior
- Edge cases (empty inputs, malformed data, boundary conditions)
- Error conditions and exception handling
- Integration points between modules
- Specific clinical scenarios with known expected outputs

**Property-Based Tests** focus on:
- Universal properties that hold for all valid inputs
- Comprehensive input coverage through randomization
- Invariants that must be maintained across operations
- Round-trip properties (serialization, parsing, encryption)
- Performance constraints across diverse inputs

Both approaches are complementary and necessary: unit tests catch concrete bugs and validate specific scenarios, while property tests verify general correctness across the input space.

### Property-Based Testing Framework

**Framework Selection:**
- **Python**: Use Hypothesis for property-based testing
- **TypeScript/JavaScript**: Use fast-check for property-based testing
- **Java**: Use jqwik for property-based testing

**Configuration:**
- Minimum 100 iterations per property test (due to randomization)
- Each property test must reference its design document property
- Tag format: `Feature: medsutra-clinical-ai-assistant, Property {number}: {property_text}`

**Example Property Test Structure:**

```python

import hypothesis

    snapshot = clinical_summarizer.generate_summary(documents)

    assert elapsed_time < 10.0


# Property 5: Information Deduplication
```
